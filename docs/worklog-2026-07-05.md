Cluster: DigitalOcean DOKS, 2 × `s-4vcpu-8gb` nodes (scaled to 3 late-session, see §8.2). Namespace focus: `yas`.

## TL;DR

The `yas` app was down (product/tax/order/customer/inventory crashlooping). Root
cause was **Postgres out of disk** (WAL bloat), not memory as first assumed.
Fixed the disk, freed cluster memory by removing a stale duplicate deployment,
recovered all 15 core services, restored metrics observability, and completed
the Service Mesh deliverables (AuthorizationPolicy + Kiali + allow/deny/retry
tests).

## 1. Root-cause analysis (measured, not assumed)

Initial symptom: `product`, `tax`, `order`, `customer`, `inventory`, `media`
crashlooping with Postgres `Connection reset` during SSL handshake. First
hypothesis was memory (nodes at 96–99% memory *requests*). **That was wrong.**

Evidence corrected it:
- Postgres log: `FATAL: could not write lock file "postmaster.pid": No space left on device` → Postgres was **down**, not slow.
- Kubelet volume stats: `pgdata` PVC **20.02 GiB / 21 GiB = 95% full**.
- `wal_level=logical` + checkpoint timeline `575` → **WAL bloat from Debezium logical replication** (CDC slot retaining WAL while the connector was disrupted).

Chicken-and-egg: WAL could not be reclaimed via SQL because Postgres would not
start with a full disk. So expansion had to come first.

### Incident during recovery (documented for honesty)
Rollout-restarting 5 JVMs at once briefly pushed node `3cw2nm` into `NotReady`
(kubelet stall) — it hosts Postgres + istiod. This cascaded:
`konnectivity-agent` (DO control-plane tunnel) CrashLooped → API-server→istiod
sidecar-injector webhook timed out (`/inject: context deadline exceeded`) →
new product/order pods failed to **create** (`ReplicaFailure: FailedCreate`).
Removing `yas-developer` freed enough memory that konnectivity recovered, the
webhook succeeded, and product/order created and reached `2/2`. Lesson: on this
cluster, restart heavy workloads **one at a time**.

### 8.1 storefront-ui / backoffice-ui crashloop
- Symptom: both UIs crashlooping (17 / 15 restarts), liveness/readiness probes failing (`HTTP 500` / `context deadline exceeded`). `storefront.yas.local.com` returned `000`.
- Root cause: the shared `ui` chart shipped a `128Mi` memory **limit** plus default tight probes (`timeout 1s`, `failureThreshold 3`) — too small/strict for a Next.js cold start (OOM + premature kill).
- Fix (chart + live patch):
  - `k8s/charts/ui/values.yaml`: memory limit `128Mi→512Mi`, cpu limit `200m→500m`, request `64Mi/10m→128Mi/25m`; added generous `livenessProbe`/`readinessProbe` defaults (`initialDelay 45/20s`, `timeout 5s`, `failureThreshold 6`).
  - `k8s/charts/ui/templates/deployment.yaml`: made both probes tunable from values (keep hardcoded `httpGet / :http`, merge the rest via `toYaml`).
  - Live `kubectl patch` on both `yas` deployments to apply immediately.
- Result: new pods `2/2`, **0 restarts** after 19m. `storefront` 200, `backoffice` 302→login.

### 9.2 Root-cause: Postgres out of disk (WAL bloat)
```bash
# Postgres pod status + the decisive log line
kubectl get pods -n postgres
kubectl logs -n postgres postgresql-0 -c postgres | grep -i 'no space\|space left\|FATAL'
#   -> FATAL: could not write lock file "postmaster.pid": No space left on device

# Measure the PVC fill level WITHOUT exec (exec was blocked by the shared-cluster
# guard). Kubelet's stats-summary API reports volume usage:
kubectl get --raw "/api/v1/nodes/<node>/proxy/stats/summary" \
  | python3 -c 'import sys,json; d=json.load(sys.stdin);
[print(v["name"], round(v["usedBytes"]/1e9,2),"/",round(v["capacityBytes"]/1e9,2),"GB")
 for p in d["pods"] for v in p.get("volume",[]) if "pgdata" in v.get("name","")]'
#   -> pgdata 20.02 / 21.0 GB = 95% full

# Confirm WAL is the bloat source (logical replication for Debezium CDC)
kubectl get postgresql -n postgres            # Zalando CR
kubectl get pvc -n postgres
```

### 9.3 Fix: expand the Postgres volume online (Zalando operator)
DO block storage has `allowVolumeExpansion=true`, so this is an online resize —
no downtime, no data loss. Edit the **CR**, not the PVC directly (the operator
owns the PVC and would fight a manual edit):
```bash
# Patch the Zalando postgresql CR: spec.volume.size 20Gi -> 30Gi
kubectl patch postgresql -n postgres <cluster-name> --type merge \
  -p '{"spec":{"volume":{"size":"30Gi"}}}'

# Operator resizes the PVC + grows the filesystem; watch it happen
kubectl get pvc -n postgres -w
kubectl get postgresql -n postgres -w        # STATUS -> Running

# Once Postgres is writable again, reconnect the DB-backed services
kubectl rollout restart deployment -n yas product tax order customer inventory
kubectl rollout status  deployment -n yas product      # repeat per svc
```
> Lesson learned: restart heavy JVMs **one at a time**. Restarting 5 at once
> stalled a node's kubelet into `NotReady`, which took down `konnectivity-agent`
> and the istiod sidecar-injector webhook (`/inject: context deadline exceeded`).

### 9.4 Reclaim WAL / verify the Debezium slot
```bash
# Slot active + how much WAL it is retaining, then force a checkpoint
kubectl exec -n postgres postgresql-0 -c postgres -- psql -U postgres -c \
"SELECT slot_name,active,
        pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(),restart_lsn)) AS retained
   FROM pg_replication_slots; CHECKPOINT;"
```

### 9.8 Service mesh — run the allow/deny/retry tests
Traffic MUST originate from a **sidecar-captured** container so Envoy wraps it in
mTLS. `curl` from inside the `istio-proxy` container runs as UID 1337, bypasses
capture, and a STRICT peer resets it (you get `000`). So use dedicated injected
clients:
```bash
# Sidecar-injected curl pods + httpbin retry backend
kubectl apply -f k8s/istio/demo/mesh-test-clients.yaml
kubectl apply -f k8s/istio/demo/retry-demo-httpbin.yaml
kubectl wait --for=condition=ready pod -n yas -l app=mesh-tester-order --timeout=120s

# SCENARIO 1 — ALLOW (SA=order): reaches app -> 404 (not a product route), NOT 403
kubectl exec -n yas mesh-tester-order -c tester -- \
  curl -s -o /dev/null -w '%{http_code}\n' http://product.yas.svc.cluster.local/

# SCENARIO 2 — DENY (SA=default): RBAC blocks it -> "RBAC: access denied" / 403
kubectl exec -n yas mesh-tester-default -c tester -- \
  curl -s http://product.yas.svc.cluster.local/
kubectl exec -n yas mesh-tester-default -c tester -- \
  curl -s -o /dev/null -w '%{http_code}\n' http://product.yas.svc.cluster.local/

# The matching Envoy RBAC deny access-log line on the product side
kubectl logs -n yas deploy/product -c istio-proxy | grep rbac_access_denied | tail -1

# SCENARIO 3 — RETRY: 1 client call -> 4 httpbin hits (1 original + 3 retries)
kubectl exec -n yas mesh-tester-order -c tester -- \
  curl -s -o /dev/null -w '%{http_code}\n' http://httpbin.yas.svc.cluster.local:8000/status/500
kubectl logs -n yas deploy/httpbin | grep -c '/status/500'

# SCENARIO 4 — mTLS STRICT: plaintext from a non-captured container is reset
kubectl exec -n yas deploy/tax -c istio-proxy -- \
  curl -s http://product.yas.svc.cluster.local/     # -> connection reset

# Cleanup after the demo (restore normal browsing)
kubectl delete -f k8s/istio/demo/mesh-test-clients.yaml
kubectl delete -f k8s/istio/demo/retry-demo-httpbin.yaml
```
