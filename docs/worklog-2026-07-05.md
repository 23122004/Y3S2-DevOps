# Work Log — 2026-07-05: Cluster recovery, resource optimization, service mesh

Cluster: DigitalOcean DOKS, 2 × `s-4vcpu-8gb` nodes (scaled to 3 late-session, see §8.2). Namespace focus: `yas`.
Kubeconfig used: `teammate-kubeconfig.yaml`.

## TL;DR

The `yas` app was down (product/tax/order/customer/inventory crashlooping). Root
cause was **Postgres out of disk** (WAL bloat), not memory as first assumed.
Fixed the disk, freed cluster memory by removing a stale duplicate deployment,
recovered all 15 core services, restored metrics observability, and completed
the Service Mesh deliverables (AuthorizationPolicy + Kiali + allow/deny/retry
tests).

## 1. Root-cause analysis (measured, not assumed)

Initial symptom: `product`, `tax`, `order`, `customer`, `inventory`, `media`
crashlooping with Postgres `Connection reset` during SSL handshake. First
hypothesis was memory (nodes at 96–99% memory *requests*). **That was wrong.**

Evidence corrected it:
- Postgres log: `FATAL: could not write lock file "postmaster.pid": No space left on device` → Postgres was **down**, not slow.
- Kubelet volume stats: `pgdata` PVC **20.02 GiB / 21 GiB = 95% full**.
- `wal_level=logical` + checkpoint timeline `575` → **WAL bloat from Debezium logical replication** (CDC slot retaining WAL while the connector was disrupted).

Chicken-and-egg: WAL could not be reclaimed via SQL because Postgres would not
start with a full disk. So expansion had to come first.

## 2. Actions taken

| # | Action | Result |
|---|---|---|
| 1 | Expand Postgres volume via Zalando CR `spec.volume.size` 20Gi→**30Gi** (DO online expand, +~$1/mo) | PVC resized, filesystem grown, Postgres writable again |
| 2 | `kubectl rollout restart` the 5 DB-backed deployments | Fresh pods connected to healthy Postgres |
| 3 | Delete stale **`yas-developer`** namespace (96 pods, 81 failed, a full duplicate YAS — recreatable via `cd-developer.yml`) | Freed ~1.1 GB + reduced istiod/webhook load |
| 4 | Fix `swagger-ui` OOM (rc=137): memory limit 64Mi→**192Mi**, liveness timeout 1s→5s (live patch + Helm values `k8s/charts/swagger-ui/values.yaml`) | swagger-ui `2/2` |
| 5 | Confirm Debezium connector `RUNNING` → active slot advances, WAL recycles | pgdata stable at 20.2G / 31.6G (64%) |

### Incident during recovery (documented for honesty)
Rollout-restarting 5 JVMs at once briefly pushed node `3cw2nm` into `NotReady`
(kubelet stall) — it hosts Postgres + istiod. This cascaded:
`konnectivity-agent` (DO control-plane tunnel) CrashLooped → API-server→istiod
sidecar-injector webhook timed out (`/inject: context deadline exceeded`) →
new product/order pods failed to **create** (`ReplicaFailure: FailedCreate`).
Removing `yas-developer` freed enough memory that konnectivity recovered, the
webhook succeeded, and product/order created and reached `2/2`. Lesson: on this
cluster, restart heavy workloads **one at a time**.

## 8. Late-session follow-up (UI crashloop + 3rd node)

Second live pass after the recovery above surfaced two more items; both fixed.

### 8.1 storefront-ui / backoffice-ui crashloop
- Symptom: both UIs crashlooping (17 / 15 restarts), liveness/readiness probes failing (`HTTP 500` / `context deadline exceeded`). `storefront.yas.local.com` returned `000`.
- Root cause: the shared `ui` chart shipped a `128Mi` memory **limit** plus default tight probes (`timeout 1s`, `failureThreshold 3`) — too small/strict for a Next.js cold start (OOM + premature kill).
- Fix (chart + live patch):
  - `k8s/charts/ui/values.yaml`: memory limit `128Mi→512Mi`, cpu limit `200m→500m`, request `64Mi/10m→128Mi/25m`; added generous `livenessProbe`/`readinessProbe` defaults (`initialDelay 45/20s`, `timeout 5s`, `failureThreshold 6`).
  - `k8s/charts/ui/templates/deployment.yaml`: made both probes tunable from values (keep hardcoded `httpGet / :http`, merge the rest via `toYaml`).
  - Live `kubectl patch` on both `yas` deployments to apply immediately.
- Result: new pods `2/2`, **0 restarts** after 19m. `storefront` 200, `backoffice` 302→login.

### 8.2 3rd node added — memory pressure resolved
- Prior state: node `3cw2nm` at **98% memory requests** — no slack for redeploys.
- Action: `doctl kubernetes cluster node-pool update 01333b63-… b47cdbd4-… --count 3` (pool `devops-project02-large`, `s-4vcpu-8gb`, 2→3).
- New node `3cvc0g` joined `Ready`, sits at **8% memory** — ample headroom. Old nodes 89% / 95% (existing pods stay put; redeploys land on the fresh node).
- This also unblocks reviving full observability (section 4) on the new node if wanted.

**Apply the UI fix live** — strategic merge patch (NOT `--type merge`; that one
demands the full container list and errors with
`spec.template.spec.containers[0].image: Required value`). Default strategic
merge keys the container array by `name`, so a partial patch works:
```bash
kubectl patch deployment storefront-ui -n yas -p '{
  "spec":{"template":{"spec":{"containers":[{
    "name":"storefront-ui",
    "resources":{"requests":{"cpu":"25m","memory":"128Mi"},
                 "limits":{"cpu":"500m","memory":"512Mi"}},
    "livenessProbe":{"initialDelaySeconds":45,"periodSeconds":20,"timeoutSeconds":5,"failureThreshold":6},
    "readinessProbe":{"initialDelaySeconds":20,"periodSeconds":15,"timeoutSeconds":5,"failureThreshold":6}
  }]}}}}'
# repeat for backoffice-ui (change both name occurrences)

# Watch the rollout land
kubectl rollout status deployment/storefront-ui -n yas
kubectl get pods -n yas -w | grep -E 'storefront-ui|backoffice-ui'
```
(The same values are baked into `k8s/charts/ui/{values.yaml,templates/deployment.yaml}` so the next Helm sync keeps them.)

**Add the 3rd node with doctl**
```bash
# 1. Find the cluster ID
doctl kubernetes cluster list --format ID,Name,Status,RegionSlug --no-header

# 2. Find the node-pool ID + current count
doctl kubernetes cluster node-pool list <CLUSTER_ID> --format ID,Name,Size,Count

# 3. Scale the pool 2 -> 3
doctl kubernetes cluster node-pool update <CLUSTER_ID> <POOL_ID> --count 3

# 4. Confirm the new count
doctl kubernetes cluster node-pool get <CLUSTER_ID> <POOL_ID> --format Name,Count

# 5. Wait for the new droplet to register + go Ready (poll ~2-4 min)
for i in $(seq 1 60); do
  n=$(kubectl get nodes --no-headers | grep -c ' Ready ')
  [ "$n" -ge 3 ] && { echo "3 nodes Ready"; break; }
  sleep 15
done
kubectl get nodes
```
